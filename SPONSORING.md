# Sponsor this project

Hermes Figma Prompt Hub is maintained as an external integration for Figma-based prompt design, validation, and future Hermes MCP workflows.

Sponsorship helps fund:

- Figma API importer work.
- MCP server implementation.
- Prompt schema and validation improvements.
- Documentation and examples for Hermes users.
- Maintenance, issue triage, and contributor support.

## Recommended option: OpenCollective

OpenCollective is the recommended funding path for this project because it is transparent and project-oriented.

Use it when:

- A company or team wants a visible invoice-friendly sponsorship path.
- The project may later support multiple maintainers.
- You want public budgets, expenses, and recurring sponsor records.
- Sponsorship should support the project rather than only one personal account.

Suggested setup:

1. Create an OpenCollective page for `hermes-figma-prompt-hub`.
2. Add a short project description and link back to this repository.
3. Publish the roadmap from `ROADMAP.md` as the funding goal.
4. Add recurring tiers for individuals and organizations.
5. Use OpenCollective updates for monthly progress notes.

Link:

https://opencollective.com/hermes-figma-prompt-hub

## Other donation options

### GitHub Sponsors

GitHub Sponsors is the lowest-friction option for developers already using GitHub. It works well for individual recurring sponsorships and makes the sponsor button visible directly on the repository.

Use it when:

- Sponsors are mostly developers.
- You want sponsorship close to issues, pull requests, and commits.
- You are comfortable with funding tied to the maintainer account.

Link:

https://github.com/sponsors/jozrftamson

### Buy Me a Coffee

Buy Me a Coffee is useful for casual one-time support. It is easy for non-technical supporters and works well for small thank-you donations after a useful feature, fix, or release.

Use it when:

- A supporter wants a quick one-time contribution.
- You want a simple public support link.
- The amount is small and does not need project-level governance.

Suggested link:

https://www.buymeacoffee.com/jozrftamson

### Ko-fi

Ko-fi is useful for one-time or lightweight recurring support. It can be positioned as a friendly way to support maintenance, documentation, and small improvements.

Use it when:

- Supporters prefer small recurring tips.
- You want a casual support option next to more formal sponsorship.
- You want to avoid making every sponsor interaction feel enterprise-oriented.

Suggested link:

https://ko-fi.com/jozrftamson

### Direct custom sponsorship

Direct custom sponsorship is for organizations that need a specific invoice, procurement process, or funded maintenance agreement. Keep this separate from issue comments so technical design decisions remain public.

Use it when:

- A company needs custom paperwork.
- Work requires a defined scope and timeline.
- You need to discuss maintenance expectations before accepting funds.

Recommended process:

1. Ask the sponsor to open a public issue for the technical need.
2. Move payment, invoicing, and contract details to private communication.
3. Keep the final technical scope public in the issue.
4. Avoid promising private-only features in the open-source repo.

## Sustainability planning

### Monthly maintenance goals

- Review new issues and pull requests weekly.
- Publish one sponsor update per month.
- Keep `ROADMAP.md` current.
- Maintain at least one `good first issue`.
- Run validation before each release or roadmap update.

### Funding priorities

1. Keep the project installable and documented.
2. Build the Figma importer.
3. Stabilize the MCP server.
4. Add tests and fixtures.
5. Support contributors with reviews and issue triage.

### Budget model

Use transparent categories:

- Maintenance: issue triage, reviews, releases.
- Development: importer, MCP tools, validation.
- Documentation: examples, setup guides, tutorials.
- Infrastructure: package publishing, CI, test data.

### Governance guardrails

- Sponsorship does not buy merge rights.
- Sponsored work should be tracked in public issues.
- Security-sensitive reports can be private until fixed.
- Public roadmap decisions should remain technical and community-oriented.
- If multiple maintainers join, prefer OpenCollective for shared project funding.
